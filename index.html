Areas de GIT

Working Directory

El area de trabajo corresponde a la carpeta donde creamos, eliminamos y realizamos las modificaciones de los archivos de nuestro proyecto. Casi nunca utilizaremos la consola en dicha area, a excepcion de la creacion del Repositorio al utilizar el comando git init o querramos clonar un repositorio remoto.

Staging Area (INDEX)

El area de Control de Cambios corresponde a un archivo dentro de la carpeta .git que identifica a que archivos y carpetas les estamos haciendo seguimiento. Se utiliza para realizar las capturas de codigo (SNAPSHOT) que luego deberan ser confirmadas mediante un commit y de esta manera corroborar el estado de los mismos.

Repository (Local/Remote)

El almacen de confirmaciones se corresponde a una estructura de subcarpetas dentro de la carpeta .git que contiene las confirmaciones realizadas en forma de archivos BLOB (Binary Large Object) y se utilizan como referencia para deshacer cambios en el directorio de trabajo, asi como tambien para comparar y resolver conflictos.

Apuntadores de GIT

HEAD

Apuntador Movil que indica al usuario donde se encuentra en el historial de confirmaciones, normalmente se ubica en el ultimo commit realizado, pero podemos utilizarlo como referencia para volver hacia atras en la historia del proyecto.

BRANCH

Apuntador Dinamico que siempre se ubican sobre el ultimo commit de la linea de tiempo en la que nos encontremos. Se utiliza para facilitar la navegacion dentro del historial y para probar caracteristicas o implementacion sin comprometer la rama principal.

TAG

Apuntado Estatico que se utiliza para identificar las versiones de nuestro proyecto y tambien facilita la navegacion dentro del historial de confirmacion. Normalmente las etiquetas se identifican como vn.s.r donde n representa la version principal, s la subversion (caracteristicas implementadas) y r las revisiones o correcciones de errores.

STASH

Apuntador de la Zona temporal de cambios que se utiliza cuando necesitamos navegar en el historial de confirmaciones o ramas y tenemos cambios pendientes, que no ameritan un commit. Estos cambios se almacenan en un commit que integra las diferencias del INDEX y la confirmacion en la que nos encontremos.
